Everything you need to upgrade a Jumbo project from v1 to v2.

:::caution
v2 includes breaking changes. After five months of real-world usage, the domain model needed corrections that couldn't be made additively. With a small user base, now was the right time to make them cleanly rather than accumulate technical debt. From v2 onward, all changes will be additive — any future removals will follow a documented deprecation plan communicated well in advance.
:::

---

## Quick start

Back up your `.jumbo/` directory, then run:

```bash
npm install -g jumbo-cli@latest
jumbo evolve --yes
```

`jumbo evolve` applies schema migrations, runs data migrations, refreshes managed configuration, and rebuilds projections in one step.

:::note
`jumbo evolve --yes` is idempotent. Re-running it after a successful upgrade produces no duplicate events or relations.
:::

## Breaking changes

### Goal completion workflow

Goals now go through a review and codification process before completion.

| v1 | v2 |
|---|---|
| `jumbo goal complete --id <id>` | `jumbo goal submit` → `jumbo goal review` → `jumbo goal approve` → `jumbo goal codify` → `jumbo goal close` |

If you have goals in `doing` status, you'll need to submit them for review before they can be closed.

### Goal statuses renamed

Goal status strings have changed. Update any scripts that match on status values.

| v1 | v2 |
|---|---|
| `to-do` | `defined` |
| `qualified` | `approved` |
| `completed` | `done` |

`jumbo evolve` migrates existing goals automatically.

### Session pause/resume removed

Session pause and resume commands have been removed. Sessions now only support `start` and `end`. Any paused sessions are automatically migrated to active status by `jumbo evolve`.

Goal pause/resume is unchanged.

### Command renamed

`jumbo goal updateProgress` has been renamed to `jumbo goal update-progress` (kebab-case). Update any scripts or automation that use the old name.

### `db rebuild` replaced by `heal`

| v1 | v2 |
|---|---|
| `jumbo db rebuild --yes` | `jumbo heal --yes` |

`heal` replays all events from the event store to reconstruct projections. Use it when your database is out of sync.

### Maintenance commands consolidated

| v1 | v2 |
|---|---|
| `jumbo repair` | `jumbo evolve --yes` |
| `jumbo db upgrade --from v1 --to v2` | `jumbo evolve --yes` |
| `jumbo dependency migrate` | `jumbo evolve --yes` |

### Project boundaries removed

The `--boundary` flag has been removed from `jumbo project init` and `jumbo project update`. Existing boundary data is dropped during migration.

## Dependency model change

In v1, dependencies modeled relationships between components. In v2, dependencies model third-party software and services (npm packages, external APIs, etc.). Component-to-component coupling is now handled by relations.

**v1** — component coupling via dependencies:

```bash
jumbo dependency add --consumer-id <id> --provider-id <id>
```

**v2** — component coupling via relations:

```bash
jumbo relation add --from-type component --from-id <id> \
  --to-type component --to-id <id> --type depends_on
```

**v2** — third-party dependencies (new):

```bash
jumbo dependency add --name better-sqlite3 \
  --ecosystem npm --package-name better-sqlite3 --version-constraint ^9.4.3
```

The legacy `--consumer-id` / `--provider-id` flags still work in v2 with a deprecation warning. They will be removed in v3.

## Rollback

If something goes wrong:

1. Restore your `.jumbo/` directory from backup
2. Reinstall the previous version: `npm install -g jumbo-cli@1`

|---|---|---|
